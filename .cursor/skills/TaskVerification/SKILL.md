# Task Verification

Перепроверка результата задачи. **Выполнять всегда после завершения задачи** (после того как код написан, файлы изменены, фича реализована).

## Порядок проверки

1. **Изначальные требования задачи**
   - Выполнено ли то, что явно просил пользователь (функционал, файлы, поведение).
   - Нет ли пропущенных пунктов из формулировки задачи.

2. **Дополнительные требования**
   - Правила из workspace rules (например `.cursor/rules/`): архитектура, Canvas/сцена, DI (Winter), MVC, события, MainWindow, Save/Config, Stage, структура папок, чеклист фичи.
   - Требования пользователя из user rules: OOP, один класс/интерфейс/структура в отдельном файле и т.д.
   - Учтены ли частые ошибки из правил (события — class, не struct; View без [Inject]; отписка от событий в OnDestroy; порядок Stage).

3. **Код-стайл**
   - Соответствие код-стайлу проекта и пользователя: без `var`, expression body для однострочных методов, summary для публичных членов, без лишних проверок (особенно для SerializeField), без early exit (логика внутри if), константы вместо «магических» чисел и строк, без Debug без запроса пользователя.
   - Линтер: при необходимости проверить изменённые файлы через read_lints.

4. **Итог**
   - Если **все** пункты 1–3 выполнены — вызвать **Unity Refresh**: `call_mcp_tool` (server `user-unityMCP`, toolName `execute_menu_item`, arguments `{"menu_path": "Assets/Refresh"}`). Так редактор подхватит изменения и запустит перекомпиляцию.
   - Если что-то не выполнено — исправить или явно сообщить пользователю, что осталось доделать; Refresh после правок выполнить по завершении.

## Когда применять

- После завершения любой задачи (фича, правка, рефакторинг, добавление меню и т.д.).
- Не пропускать шаг проверки перед финальным ответом пользователю.
